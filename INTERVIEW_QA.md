# VidMotion 面试追问 Q&A

---

## 一、简历项目描述

### 项目背景

中小广告主面临视频广告制作成本高（单条 2000–5000 元）、周期长（3–5 人天）的资源约束，而市面上 AI 视频工具（Runway、Pika）沿用文生图的「大文本框」范式，要求用户撰写复杂 prompt，对缺乏专业能力的中小店主极不友好。VidMotion 定位为「让用户不需要知道 AI 存在」的图生视频工具——上传商品图 → AI 自动分析 → 输出可投放视频。

### 核心工作

- **产品设计**：以 6 种可视化运动效果卡片替代传统 prompt 文本框，用户所见即所得，无需文字描述镜头运动；设计「做同款」功能将成功模板的 CTR 数据转化为增长飞轮起点
- **AI 策略**：面向 80% 中小广告主选择「图生微动」路线而非文生视频——不是因为技术限制，而是因为广告场景（开屏/Banner/信息流）不需要复杂叙事，3–5 秒微动即可完成注意力抓取；运动效果的可枚举性天然适合预设化产品形态
- **质量体系**：从内容准确性、动作连贯性、色彩一致性三个维度建立自动化质检评分，设定综合 ≥ 80 分可投放的阈值，并设计「自动质检 → 人工抽检校准 → 投放数据反哺」的反馈闭环
- **文档输出**：独立完成 PRD、5 个关键决策记录、模型评测标准文档

### 项目成果

- 完整可交互产品原型，6 个页面覆盖「上传→分析→配置→生成→评估→看板」全流程
- 部署上线：https://ai-video-ad-platform-five.vercel.app
- GitHub: https://github.com/jiujiuchi/ai-video-ad-platform
- 体现 AI PM 核心能力：用户需求洞察、AI 交互范式创新、评估体系设计、技术约束下的产品决策

## 二、技术栈选型解释（PM 视角，不需要会写代码）

### 为什么选 Next.js？（而不是纯 HTML 或 Vue）

**一句话**：Next.js 是 AI 产品原型开发的事实标准，Vercel 一键部署让 PM 也能独立交付可访问的产品。

| 考量维度 | 选择理由 |
|----------|----------|
| **快速原型** | App Router 的文件系统路由——创建文件夹 = 创建页面，零配置。PM 不需要学路由配置 |
| **部署成本** | Vercel 免费一键部署，每次 git push 自动更新。不需要买服务器、配 Nginx |
| **生态适配** | Vercel AI SDK 原生支持 OpenAI/Anthropic/Replicate 等 AI API 的流式调用（Streaming），P2 接入真实 AI 时无需重构 |
| **团队协作** | React 生态在国内开发者中普及率最高，后续交给工程团队时沟通成本最低 |

### 为什么选 Tailwind CSS？

**一句话**：PM 做原型最快的方式——直接在 HTML 标签上写样式，不用在两个文件之间跳来跳去。

- 传统 CSS：需要给每个元素命名 class → 写 CSS 文件 → 反复切换文件
- Tailwind：`rounded-lg bg-white p-4` 直接写在标签上，所见即所得
- 打包时自动删除未使用的样式 → 线上加载速度快

### 为什么选 Framer Motion？

**一句话**：做动画最简单、代码最少的 React 库。

这个项目有 3 个动画场景：
1. 页面切换动画（上传→配置→进度→结果）
2. 生成进度条的实时推进
3. 质检评分圆环的填充动画

Framer Motion 用声明式语法（`<motion.div animate={{ opacity: 1 }} />`），几行代码就能做出流畅的交互动效。用纯 CSS 动画实现同等效果代码量会翻 3–5 倍。

### 为什么选 Recharts？

**一句话**：React 生态中最轻量的图表库，适合 2–3 个图表场景，不需要引入重量级的 ECharts。

- ECharts 功能强大但打包体积大（~1MB），对于只展示趋势折线图和对比柱状图的看板来说过度设计
- Recharts 每个图表组件独立引入，只加载用到的部分（~50KB）

### Next.js 是什么？

Next.js 是 Vercel 公司开发的 React 框架。你可以理解为 React 的「增强版」——它在 React 基础上内置了路由、服务端渲染、静态生成、API 路由等功能。类比：React 是毛坯房，Next.js 是精装房。

作为 PM 你只需要记住：**它是当前 AI 产品开发最主流的框架**（ChatGPT 竞品中多数前端都用 Next.js），原因是它和 Vercel 部署深度绑定，且对 AI API 的流式调用支持最好。

### CTR 是什么？

CTR = Click-Through Rate = 点击率 = 看到广告后点击的人数 / 看到广告的总人数

- 例如：1000 人看到你的广告，40 人点击，CTR = 4%
- CTR 是衡量广告「吸睛程度」的最基础指标——它不代表最终转化，但代表创意的吸引力
- 在 VidMotion 中，模板市场展示「CTR +18%」是指使用这个模板的 AI 视频相比同场景传统视频的 CTR 提升了 18%（例如从 3.5% 提升到 4.13%）

CTR 数据在 VidMotion 中有两个作用：
1. **获客转化**：中小广告主看到「CTR +27%」更有动力点击「做同款」
2. **增长飞轮**：高 CTR 模板被更多人使用 → 产生更多投放数据 → 筛选更优质模板

## 三、可能的追问及回答思路

### Q1: 为什么不做文生视频？文生视频不是更灵活吗？

**回答思路**：

「这个问题分两层看——技术层和产品层。

技术层面，我们评估时（2023 年）文生视频在主体一致性上还不够稳定，用户写了一堆描述但生成出来可能完全不是自己的产品。图生视频因为有原始图片作为锚点，主体保持率高得多。

产品层面更重要：中小广告主不需要「灵活」。他们只需要「把我的产品展示出来」。文生视频的灵活性意味着更多的不确定性和试错成本——每次生成耗时几十秒到几分钟、消耗大量 token。我们选择在广告场景中把运动类型限制在 6 种预设中，不是因为技术做不了，而是因为产品场景不需要更多。

如果今天重新选型（2026 年），可灵、Luma 的图生视频能力已经大幅提升，我会重新做技术评估，但产品形态（短时微动 + 预设效果）不会变——因为这是场景决定的，不是技术决定的。」

---

### Q2: 如果用户对某一种运动效果的需求特别强烈，但你们不支持，怎么办？

**回答思路**：

「首先，广告场景中的镜头运动类型是有限的（推/拉/摇/移/跟/升/降），我们覆盖了 6 种核心运动，已经满足绝大多数广告场景。

其次，我们在高级设置中保留了 prompt 文本框入口——它不是默认打开的，但对于那些确实有特殊需求的用户，他们可以手动修改 AI 生成的 prompt 描述。这是一个「渐进式复杂度」的设计：80% 用户永远不需要打开高级设置，20% 的进阶用户有出口可以深入。

最后，如果运营数据发现某个垂直行业（比如珠宝）对某种我们没有覆盖的运动效果（如「微距旋转」）有持续需求，我们可以新增一个运动效果卡片。产品机制本身支持平滑扩展。」

---

### Q3: 质检评分的阈值是怎么定的？为什么综合 80 分可投放？

**回答思路**：

「阈值不是拍脑袋确定的。在原始项目中，我们是这么校准的：

1. 先跑一批视频（比如 100 条），让运营和测试团队人工打分
2. 将自动化评分的结果与人工评分做对照，找到「80% 人工通过率」对应的自动化评分线
3. 那条线差不多就在 80 分

所以「综合 ≥ 80 可投放」的含义是：**在这个阈值下，80% 以上经人工审核的视频被认为是合格的**。

阈值不是一成不变的——我们设计了每月校准机制，根据投放数据的实际表现（CTR、用户投诉率、素材拒审率）来动态调整。如果某个月投放效果下降，可能需要上调阈值；如果通过率过低导致供给不足，可能需要下调或增加人工抽检比例。这是 AI 产品运营的日常。」

---

### Q4: 你在这个项目里写的代码量有多少？PM 要做原型到这种程度吗？

**回答思路**：

「这个项目的代码确实是我写的，但我的目的不是证明我能写代码——而是证明我能把产品想法变成可被他人体验的东西。

PM 做原型的程度取决于你的产品决策需要什么粒度的验证。有些决策在白板上就能做（比如信息架构），有些在 Figma 里做（比如交互流程），但这个项目涉及 AI 交互范式的创新——用可视化卡片替代 prompt 文本框——这种交互创新的体验在静态原型里验证不了。你需要真正上传一张图、看到运动效果卡片亮起来、点击生成看到进度动画，才能判断这个设计是否成立。

我用 Next.js 做这个原型，一方面是因为它的部署成本极低（push 代码即上线），另一方面是因为真实 AI 视频生成 API 的接入路径最短（Vercel AI SDK 原生支持流式调用）。当 P2 需要接 Replicate 或可灵 API 时，不需要重构。」

---

### Q5: 「做同款」不会导致广告素材同质化吗？平台方不会限制吗？

**回答思路**：

「这是一个很好的问题，分两方面看。

第一，「做同款」复制的是运动方式和镜头节奏，不是画面内容。两个卖家——一个卖运动鞋、一个卖保温杯——用同一个「白底商品展示」模板，最终生成的视频画面完全不同，因为原始图片不同。这就像两个人用同一个 Instagram 滤镜，照片本身不会变一样。

第二，同质化风险确实存在，我们有两个防御机制：
- 如果某个模板被过度使用，同一广告位可能出现「运动方式撞车」。产品层面可以设置模板热度上限，或者引入运动方式的微变体（比如同一个镜头缓推方向分左往右 / 右往左 / 斜向）
- 广告平台（如巨量引擎、腾讯广告）会对同质化素材降低出价或限制曝光分配。这反而会产生一个产品需求：当用户选择高频模板时，主动提示「该模板当前使用人数较多，建议尝试相似的变体版本」」

---

### Q6: 这个项目用户增长怎么做？怎么获客？

**回答思路**：

「这个项目目前的定位是 PM 作品集项目，还没有真实的用户增长。但是如果我负责这个产品的增长，我会关注三个渠道：

1. **产品内增长**：模板市场本身就是获客工具。「CTR +27%」这类数据是中小广告主无法拒绝的信号。他们可能不是搜索 VidMotion 来的，而是在同行群/朋友圈看到别人分享的效果截图。
2. **平台生态获客**：如果 VidMotion 生成的视频投放在巨量引擎/腾讯广告上，可以在视频末尾加「由 VidMotion 生成」的水印或标签页。广告优化师之间信息流通效率很高——一个人用了觉得好，会自发推荐给同事。
3. **内容营销**：在中小电商聚集的社区（小红书商家号、抖音电商创作者社区）发对比内容——「同一条商品图，传统视频 vs VidMotion AI 视频，CTR 差多少」。真实数据对比是这个群体最有效的转化内容。

核心逻辑：不是我找用户，而是让用户在投放广告的过程中发现 VidMotion——产品即获客渠道。」

### Q7: 为什么不直接接入 GPT-4o 或可灵 API？Mock 数据的原型说服力够吗？

**回答思路**：

「这个项目 P0（当前阶段）是验证产品交互和用户路径，不是验证 AI 模型能力。用 Mock 数据是刻意的选择：

1. **成本约束**：AI 视频生成 API 调用一次几毛到几块钱不等。频繁调试产品交互时，Mock 数据 0 成本、即时响应。如果每次改布局都要等 30 秒生成并花几块钱，PM 的迭代效率会大幅降低。
2. **关注点隔离**：用 Mock 数据可以让我先回答「用户需要什么交互」再回答「AI 能不能做到」。如果一开始就接真实 API，很容易被模型的能力缺陷（比如偶尔生成出奇怪的画面）带偏，去优化模型质量而不是优化产品体验。
3. **P2 接入成本低**：当前的组件接口（ImageUploader → analyzeImage → motionStyles → generate → qualityScore）已经设计为与真实 API 对接。替换 Mock 函数就是把一个返回固定 JSON 的函数换成 fetch API 调用——产品逻辑不需要改。

原型在这个阶段的「说服力」不来自于视频真的好看，而是来自于：面试官能顺着链路走完「上传→分析→选择→生成→评分」的完整流程，理解这个产品的价值主张。如果我丢给你一行 API key 让你自己跑，大概率你卡在第一步就不会继续了。」

### Q8: 如果给你三个月时间和一个小型工程团队，你会怎么推进这个项目？

**回答思路**：

「第一个月：验证核心假设。

最重要的假设不是『AI 能不能生成视频』——这个已经被可灵和 Luma 验证过了——而是『中小广告主是否接受预设运动效果而不是自由 prompt』。我会找 5–10 个中小广告主做可用性测试：给他们上传界面，看他们能否不借助指引完成生成；看他们是否会主动打开高级设置；询问他们对运动效果的命名是否能理解（比如『镜头缓推』vs『Ken Burns 效果』）。同时工程团队接入 1–2 个真实 AI 视频生成 API，跑通图生视频链路。

第二个月：MVP 上线，监控核心指标。

重点看三个指标：（1）生成成功率——上传后到可投放视频的端到端转化率，（2）质检评分分布——评估接入真实 API 后的质量波动，（3）NPS——用户是否愿意推荐给别人。工程团队在这个月主要做稳定性优化和错误处理——AI API 挂了怎么办、生成超时怎么办、质量不合格怎么给用户反馈。

第三个月：增长实验。

围绕『做同款』做 A/B 测试——展示 CTR 数据 vs 不展示数据，转化率有没有差异；尝试在生成结果的视频末尾自动添加水印引流；在模板市场增加品类筛选和个性化推荐（基于用户的上传图片品类历史）。」

## 四、面试高频概念速记

| 概念 | 一句话解释 | 在这个项目哪里用到 |
|------|-----------|-------------------|
| CTR | 点击率 = 点击人数 / 曝光人数 | 模板市场的 CTR 提升数据、效果看板的场景 CTR 对比 |
| ECPM | 千次展示期望收入，衡量广告位的变现效率 | 背景说明：视频广告 ECPM 高于图片，所以中小广告主需要视频 |
| MVP | Minimum Viable Product，最小可行产品 | P0 6 个页面的功能范围定义（PRD 中标注了 V1/V2 范围） |
| 增长飞轮 | 自我强化的增长循环，每个阶段为下一阶段提供燃料 | 「做同款」→ CTR 数据 → 模板收录 → 更多人用 → 更多数据 |
| 质检闭环 | 自动化评分 + 人工校准 + 数据反馈的持续改进循环 | 三维评分体系 + 人工抽检 + 投放数据反哺 |
| 渐进式复杂度 | 默认简单、逐步暴露高级功能的设计策略 | 默认选模板即可生成，高级设置折叠面板中保留 prompt 编辑 |
| RAG | 检索增强生成 = 先查资料再生成答案，减少幻觉 | 未直接使用，但在 P2 规划中可用于模板推荐（基于用户历史偏好检索匹配模板） |
| Fine-tuning | 微调 = 在预训练模型上用小数据集做针对性训练 | 未直接使用，但在 P2 规划中可用于优化特定行业（如美妆）的生成效果 |
